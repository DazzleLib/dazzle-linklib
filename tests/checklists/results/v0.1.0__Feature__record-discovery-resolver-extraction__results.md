**Design Note -- `resolve_target` and tool-created records:**
When the tool creates a record with `target_representations: {}` (no `original_path` key in target_representations), `get_locators()` returns no path locators. `resolve_target` returns `None` even when the target file exists. The target is in `link.target_path`, but `get_locators()` only iterates `target_representations` (not `link.target_path` directly). This is expected behavior given the library's design contract, but callers who create records via the tool without populating `original_path` in `target_representations` will get `None` from `resolve_target` until rebase or an explicit `add_locator` is called. Not a bug -- documented as a usage note.

## Bug / Gap Report

### GAP-1: `rebase` cannot process polyglot (embedded-script) `.dazzlelink` files

**Severity:** Medium (affects all executable `.dazzlelink` files when `rebase` is called on a directory containing them)

**File:** `dazzle_linklib/discovery.py`, function `rebase`, line 159.

**Root cause:** `rebase` opens `.dazzlelink` files with `json.load(f)` directly. This works for plain JSON and legacy flat records, but fails on the polyglot executable-script format where JSON follows a `# DAZZLELINK_DATA_BEGIN` marker line. The `json.JSONDecodeError` (a `ValueError` subclass) is caught by `except (OSError, ValueError)` and the record is silently placed in `result['errors']` with message `Expecting value: line 1 column 1 (char 0)`.

`DazzleLinkData.from_file` handles all three formats correctly. `rebase` should call it (or share the parsing logic) instead of calling `json.load` directly.

**Repro:**
```python
import os, tempfile, json
from dazzle_linklib import rebase

d = tempfile.mkdtemp()
real = os.path.join(d, 'target.txt')
open(real, 'w').write('content')
p = os.path.join(d, 'poly.dazzlelink')
record_data = {'schema_version': 1, 'link': {'original_path': real, 'target_path': real, 'type': 'symlink', 'target_representations': {'original_path': real, 'relative_path': 'STALE.txt'}}, 'config': {'default_mode': 'info'}}
content = '#!/usr/bin/env python3\nprint(1)\n# DAZZLELINK_DATA_BEGIN\n' + json.dumps(record_data) + '\n'
open(p, 'w', encoding='utf-8').write(content)
result = rebase(d)
print(result)
# Output: {'changed': [], 'unchanged': [], 'errors': [{'file': '...poly.dazzlelink', 'error': 'Expecting value: line 1 column 1 (char 0)'}]}
```

**Fix direction:** In `rebase`, replace `json.load(f)` with `DazzleLinkData.from_file(dl_path)` and read `.data` from the result, then save back with `DazzleLinkData(data).save_to_file(dl_path)`. This converts the file to plain JSON on write (acceptable -- saving a rebase write strips the script wrapper, which is a separate behavior decision the library should document).

**Regression test added:** `tests/test_rebase_polyglot_gap.py` (2 tests -- documents current behavior and confirms `from_file` reads polyglot correctly). Tests pass today asserting the gap behavior; when the fix lands, flip the assertions to confirm the fix.

## Resolution (post-run, same session)

Both findings were fixed immediately rather than deferred:

1. **GAP-1 (rebase + polyglot)** — fixed in `dazzle_linklib/discovery.py`. `rebase`
   now classifies a non-JSON file via `DazzleLinkData.from_file`: polyglot
   (executable-script) records go to a new `skipped` bucket with a clear reason
   and are **left byte-for-byte intact** (the library deliberately does not
   rewrite them as plain JSON, which would strip the script wrapper — that
   regeneration is the dazzlelink CLI tool's job). Genuinely unparseable files
   still go to `errors`. The agent's suggested "write plain JSON" fix was
   rejected for exactly the wrapper-stripping reason.

2. **Design Note (resolve_target / get_locators)** — re-classified as a real
   correctness bug and fixed in `dazzle_linklib/record.py`. `get_locators()` now
   always seeds the stored `target_path` as the first `path` locator, so a record
   with `target_representations == {}` (the common shape, incl. tool-created
   records) resolves correctly. Backward-compatible (deduped when
   `original_path == target_path`).

Tests updated: `test_rebase_polyglot_gap.py` superseded by `test_rebase_polyglot.py`
